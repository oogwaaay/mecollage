### 2. ✅ 性能评分下降 - PNG 图片预加载问题
**问题**：
- 网络负载过大：4,991 KiB（约 5MB）
- `wallImage.png`: 2,509.6 KiB
- `placeholder-icon.png`: 2,056.7 KiB
- CSS 中先加载 PNG 作为回退，导致即使支持 WebP 也会先请求 PNG

**修复策略**：
1. **移除 CSS 中的 PNG 预加载**
   - 从 `ghibli-background.css` 中移除 `background-image: url('/wallImage.png')`
   - 从 `styles.css` 中移除 `background-image: url('/placeholder-icon.png')`

2. **使用 JavaScript 动态设置背景图片**
   - 在 `<head>` 中添加内联脚本，尽早检测 WebP 支持
   - 检测到 WebP 支持后，立即设置 WebP 背景图片
   - 不支持 WebP 的浏览器才加载 PNG

3. **使用 CSS 自定义属性**
   - 使用 `--placeholder-icon-url` CSS 变量
   - JavaScript 根据 WebP 支持情况动态设置变量值

**修改的文件**：
- `index.html`：添加 `<head>` 内联脚本进行早期 WebP 检测
- `src/ghibli-background.css`：移除 PNG 预加载
- `src/styles.css`：使用 CSS 变量，移除 PNG 预加载
- `src/main.js`：简化 WebP 检测逻辑（检测已在 head 完成）

**预期效果**：
- 现代浏览器（支持 WebP）：只加载 WebP 文件（~200 KiB），不加载 PNG
- 旧浏览器（不支持 WebP）：加载 PNG 文件（作为回退）
- 网络负载从 4,991 KiB 降至 ~400 KiB（节省 ~4,500 KiB）

## 技术实现细节

### WebP 检测和设置流程

```
1. <head> 内联脚本执行（页面加载早期）
   ↓
2. 检测 WebP 支持（使用 base64 测试图片）
   ↓
3. 如果支持 WebP：
   - 添加 'webp-supported' 类到 <html>
   - 设置 CSS 变量：--placeholder-icon-url = url(/placeholder-icon.webp)
   ↓
4. DOMContentLoaded 事件
   ↓
5. 设置 .ghibli-background 背景图片：
   - WebP 支持：url(/wallImage.webp)
   - 不支持：url(/wallImage.png)
   ↓
6. 初始化应用
```

### 关键代码位置

1. **早期检测**：`index.html` `<head>` 中的内联脚本
2. **CSS 变量**：`src/styles.css` 中的 `var(--placeholder-icon-url)`
3. **背景设置**：`src/main.js` 中的 DOMContentLoaded 处理
